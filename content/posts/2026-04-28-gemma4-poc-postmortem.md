## 오늘 한 일

임직원 개인 맞춤형 AI 비서 프로젝트에서 로컬 LLM 방식의 PoC를 공식 종료했다.

지난 몇 주간 Gemma4를 올려놓고 실제 업무 시나리오로 돌려봤는데, 결론부터 말하면 **현업에 쓸 수 없는 수준**이었다. 할루시네이션이 빈번하고, 처리 속도가 느리고, 결과물 품질이 클라우드 모델과 비교하면 현저히 떨어졌다. 특히 복잡한 멀티스텝 업무—보고서 작성, 회의록 정리, 전략 분석 같은—에서는 거의 쓸모가 없었다.

## 왜 로컬을 시도했나

이유는 명확했다. **보안**이다. 산업안전 데이터를 다루는 회사 특성상, 민감한 현장 데이터가 외부 서버로 나가는 것에 대한 우려가 있었다. 로컬에서 모든 걸 처리할 수 있다면 이상적이었다.

하지만 현실은 달랐다.

## 왜 포기했나

네 가지 이유:

1. **할루시네이션**: 정확해야 하는 안전 관련 데이터에서 사실을 지어내는 건 치명적이다
2. **속도**: 응답이 너무 느려서 실무자들이 쓰지 않을 게 뻔했다
3. **품질**: 클라우드 모델 대비 결과물 격차가 너무 컸다
4. **복잡 업무 불가**: 단순 Q&A는 되지만, 실제 업무 자동화에는 역부족

## 현재 채택한 구조

최신 클라우드 모델 조합으로 전환했다. 속도와 품질 모두 만족할 만한 수준이다. 보안 우려는 API 레벨의 데이터 처리 정책과 접근 제어로 대응하기로 했다.

결국 **"완벽한 보안 + 낮은 품질"보다 "합리적 보안 + 높은 품질"이 현실적인 선택**이었다.

## 다른 일

오후에는 사내 조직 구조를 정리했다. AX를 추진하려면 각 팀이 어떤 업무를 하고, 어디에 AI가 끼어들 수 있는지 파악해야 한다. 스마트서비스팀의 IoT 연동, 플랫폼개발팀의 SI 사업, 전략기획실 산하 각 팀의 역할을 맵핑했다. 이게 다음 단계—팀별 AI 비서 커스터마이징—의 기초 자료가 된다.

## 배운 것

- 로컬 LLM의 한계는 모델 크기가 아니라 **추론 품질**의 문제다. 파라미터를 아무리 늘려도 클라우드 최상위 모델의 RLHF와 후처리를 따라잡기 어렵다.
- PoC는 빠르게 죽여야 한다. 가능성에 미련을 두면 시간만 낭비한다.
- AX는 기술 선택보다 **조직 이해**가 먼저다. 어떤 팀이 어떤 업무를 하는지 모르면 AI를 어디에 넣을지도 모른다.

내일은 팀별 업무 패턴 분석을 시작할 예정이다.
